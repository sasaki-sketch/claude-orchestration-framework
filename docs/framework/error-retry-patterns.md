# エラー時リトライパターン

**作成日**: 2026-02-16
**目的**: エージェント失敗時の標準的なリトライ手順を定義

---

## 1. エラー種別と対応方針

### 1.1 エラー分類

| エラー種別 | 原因 | リトライ可否 | 待機時間 |
|-----------|------|-------------|---------|
| **レート制限** | API呼び出し過多 | ○ | 5-10分 |
| **タイムアウト** | 処理時間超過 | ○（分割後） | 即時 |
| **出力エラー** | パス誤り、権限 | ○（修正後） | 即時 |
| **コンテキスト超過** | 入力過大 | ○（分割後） | 即時 |
| **モデル制限** | Haiku能力不足 | ○（モデル変更） | 即時 |

### 1.2 リトライ判断フロー

```
エラー発生
    │
    ├─ レート制限？
    │   └─ Yes → 5-10分待機 → リトライ
    │
    ├─ タイムアウト？
    │   └─ Yes → タスク分割 → 分割タスクで再実行
    │
    ├─ 出力エラー？
    │   └─ Yes → パス/権限修正 → リトライ
    │
    ├─ コンテキスト超過？
    │   └─ Yes → 入力削減/分割 → リトライ
    │
    └─ モデル能力不足？
        └─ Yes → Sonnet/Opusに変更 → リトライ
```

## 2. エラー別リトライ手順

### 2.1 レート制限（Rate Limit）

**症状**:
```
You've hit your limit · resets 4am
```

**対応手順**:

1. **即時対応**: 残りのWaveを中断
2. **記録**: 失敗したタスクをメモ
3. **待機**: リセット時間まで待機（または5-10分）
4. **再実行**: 失敗タスクのみリトライ

**予防策**:
- 並列数を5以下に制限
- Wave間に1-2分のインターバル
- 重要タスクを先に実行

**リトライテンプレート**:
```
## リトライ: [タスク名]

前回失敗理由: レート制限
待機時間: [X]分

## 必須読み込み
[通常のspawn-template]

## タスク
[元のタスク内容]
```

### 2.2 タイムアウト（Timeout）

**症状**:
- エージェントが応答しない
- 長時間「処理中」のまま

**対応手順**:

1. **タスク分析**: 何が時間を消費したか特定
2. **分割設計**: タスク分割ガイドに従い分割
3. **再実行**: 分割後のタスクで実行

**分割例**:
```
元タスク（タイムアウト）:
「全ドキュメント（15件）をレビュー」

分割後:
├── Task A: ドキュメント1-5をレビュー
├── Task B: ドキュメント6-10をレビュー
├── Task C: ドキュメント11-15をレビュー
└── Task D: 結果統合
```

### 2.3 出力エラー（Output Error）

**症状**:
- ファイルが期待したパスにない
- 権限エラー
- 出力形式が異なる

**対応手順**:

1. **原因特定**: パス、権限、形式のどれか
2. **プロンプト修正**: 明示的なパス/形式指定を追加
3. **再実行**

**修正テンプレート**:
```
## タスク

[タスク内容]

## 出力指定（必須）

- 出力先: /Users/sasaki/projects/mutsubi-dx/[具体的パス]
- 形式: Markdown
- ファイル名: [具体的なファイル名]

※ 相対パスではなく絶対パスを使用してください
```

### 2.4 コンテキスト超過

**症状**:
- 「入力が大きすぎます」エラー
- 応答が途中で切れる

**対応手順**:

1. **入力削減**: 必要最小限のファイルに絞る
2. **タスク分割**: 処理対象を分割
3. **再実行**

**削減テンプレート**:
```
## 必須読み込み

以下のファイルのみ読んでください（他は不要）:
1. [必須ファイル1]
2. [必須ファイル2]

※ 関連ファイルは必要に応じて参照
```

### 2.5 モデル能力不足

**症状**:
- Haikuで複雑な分析が不十分
- 浅い回答、重要な観点の欠落

**対応手順**:

1. **評価**: タスク複雑度を再評価
2. **モデル変更**: Sonnet または Opus に変更
3. **再実行**

**モデル変更基準**:

| 失敗パターン | 推奨モデル |
|-------------|-----------|
| 複数ドキュメント比較が不十分 | Sonnet |
| セキュリティ観点の欠落 | Opus |
| 因果関係の分析が浅い | Sonnet |
| アーキテクチャ判断ミス | Opus |

## 3. リトライ回数の上限

### 3.1 上限設定

| エラー種別 | 最大リトライ | 上限後の対応 |
|-----------|-------------|-------------|
| レート制限 | 2回 | 翌日に延期 |
| タイムアウト | 2回 | 手動実行 |
| 出力エラー | 3回 | プロンプト見直し |
| モデル能力不足 | 1回 | Opusで実行 |

### 3.2 エスカレーション基準

以下の場合はリトライを中止し、手動対応:

```
□ 3回連続で同じエラー
□ 原因が特定できない
□ リトライコストが元タスクを上回る
□ 緊急度が高く待機できない
```

## 4. リトライログの記録

### 4.1 記録フォーマット

```markdown
## リトライログ

### [日付] [タスク名]

| 項目 | 内容 |
|------|------|
| エラー種別 | [レート制限/タイムアウト/...] |
| 発生Wave | Wave [N] |
| 対応 | [待機/分割/モデル変更/...] |
| 結果 | [成功/失敗] |
| 学び | [得られた知見] |
```

### 4.2 記録先

`/logs/retry-log.md` に追記

## 5. 予防的対策

### 5.1 事前チェックリスト

```
□ 並列数は5以下か
□ 各タスクは10分以内で完了する見込みか
□ 出力パスは絶対パスで明示したか
□ タスク複雑度に適したモデルを選択したか
□ 入力ファイル数は適切か（10件以下推奨）
```

### 5.2 Wave設計時の考慮

```
□ クリティカルなタスクを先のWaveに配置
□ 失敗の影響が小さいタスクを後のWaveに
□ 統合タスクは軽量に設計
□ Wave間にバッファ時間を確保（オプション）
```
