# mutsubi-dx Agent Teams ガイド

**作成日**: 2026-02-16
**目的**: Agent Teams機能を活用した高度なエージェント協調作業の実現

---

## 概要

Agent Teamsは複数のClaude Codeインスタンスがチームとして協調作業を行う実験的機能です。

### アーキテクチャ

```
Team Lead（リーダー）
    │
    ├── 共有タスクリスト
    │   ├── Task 1: pending
    │   ├── Task 2: in_progress (Teammate A)
    │   └── Task 3: completed
    │
    ├── Teammate A ←→ 直接通信 ←→ Teammate B
    └── Teammate C
```

### Subagents vs Agent Teams

| 項目 | Subagents | Agent Teams |
|------|-----------|-------------|
| コンテキスト | 独立（結果のみ返却） | 独立（相互通信可能） |
| 通信 | オーケストレーター経由 | 直接メッセージ可能 |
| タスク管理 | 手動（Wave方式） | 自動（共有タスクリスト） |
| 最適な用途 | 独立したタスク | 協調が必要なタスク |
| トークンコスト | 低（結果のみ返却） | 高（各インスタンス独立） |

## 有効化

### 設定済み（mutsubi-dx）

`/.claude/settings.json`:
```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  },
  "teammateMode": "in-process"

```

## 使用方法

### 1. チーム作成

自然言語でClaude Codeに指示:

```
D-1 PoCのレビューを行うエージェントチームを作成してください。
3人のチームメイトを起動:
- セキュリティ観点のレビュー担当
- パフォーマンス観点のレビュー担当
- テストカバレッジ確認担当
```

### 2. チームメイトとの対話

**In-processモード（デフォルト）**:
- `Shift+Up/Down`: チームメイト選択
- 選択後にメッセージ入力: 直接指示
- `Ctrl+T`: タスクリスト表示/非表示

### 3. タスク管理

リードがタスクを作成し、チームメイトが自動でclaimまたはリードが割り当て:

```
リード: "PRDセクション1-5のレビュータスクを作成して、各担当に割り当ててください"
```

### 4. チーム終了

```
チームメイトをシャットダウンして、チームをクリーンアップしてください
```

## mutsubi-dxでの活用シナリオ

### シナリオ1: 並列レビュー

```
チーム構成:
├── Lead: mutsubi-pm（オーケストレーター）
├── Teammate 1: セキュリティレビュー
├── Teammate 2: 設計レビュー
└── Teammate 3: 法規制レビュー

タスク:
1. PRD v1.3をレビュー
2. 発見した問題を共有タスクリストに登録
3. リードが統合レポート作成
```

### シナリオ2: 競合仮説調査

```
チーム構成:
├── Lead: mutsubi-pm
├── Teammate 1: 仮説A調査
├── Teammate 2: 仮説B調査
└── Teammate 3: 仮説C調査

プロセス:
1. 各チームメイトが異なる仮説を調査
2. 発見を直接共有し議論
3. 最も有力な仮説を特定
```

### シナリオ3: 新機能実装

```
チーム構成:
├── Lead: mutsubi-pm
├── Teammate 1: フロントエンド担当
├── Teammate 2: バックエンド担当
└── Teammate 3: テスト担当

タスク:
1. 各担当が独立して実装
2. 依存関係はタスクリストで管理
3. リードが統合とリリース判断
```

## 現行運用（Subagents）との併用

### 使い分け基準

| 状況 | 推奨 |
|------|------|
| 独立した並列タスク | Subagents（Wave方式） |
| 相互議論が必要 | Agent Teams |
| シンプルな作業 | Subagents |
| 複雑な協調作業 | Agent Teams |
| コスト重視 | Subagents |
| 品質重視 | Agent Teams |

### 移行パターン

```
現行（Subagents + Wave）:
Wave 1: 3エージェント並列 → Wave 2: 統合

Agent Teams:
チーム作成 → 自動協調 → 統合
```

## ベストプラクティス

### 1. チームサイズ

- **推奨**: 3-4チームメイト
- **最大**: 5チームメイト
- **理由**: 通信オーバーヘッドとトークンコスト

### 2. タスク粒度

- **適切**: 自己完結型、明確な成果物
- **不適切**: 過度に細かい、依存関係が複雑

### 3. Plan承認の活用

複雑なタスクでは、チームメイトにPlan承認を要求:

```
アーキテクト担当のチームメイトを起動してください。
実装前にプラン承認を必須にしてください。
```

### 4. Delegateモードの活用

リードが実装に入らないようにする:

```
Shift+Tab でDelegateモードに切り替え
→ リードは調整のみ、実装はチームメイトが担当
```

#### Delegateモード実践例

**例1: PRDレビューの委譲**
```
[Delegateモードで起動]
リード: "PRD v1.3のセクション1-3をレビューしてください"
→ チームメイトがレビュー実行
→ リードは進捗監視のみ
→ 完了後、リードが結果を統合
```

**例2: 並列調査の委譲**
```
[Delegateモードで起動]
リード: "AppSheetの認証方式を3つ調査してください"
→ 各チームメイトが1方式ずつ調査
→ リードは介入せず待機
→ 完了後、リードが比較表を作成
```

**Delegateモードの利点**:
- リードのコンテキスト消費を抑制
- チームメイトの自律性を促進
- 並列作業の効率化

### 計測対象

| メトリクス | 測定方法 | 目標値 |
|-----------|---------|--------|
| チーム起動成功率 | 成功/試行 | 90%以上 |
| タスク完了率 | 完了/作成 | 95%以上 |

### 振り返りチェックリスト

各Agent Teamsセッション終了時:

```
□ チームサイズは適切だったか（3-4人）
□ 直接通信は有効活用されたか
□ Subagentsで代替可能だったか
□ コストは想定内だったか
□ 学びを shared-learnings.md に記録したか
```

## 注意事項・制限

| 制限 | 説明 |
|------|------|
| セッション再開 | `/resume`でチームメイトは復元されない |
| 1セッション1チーム | 同時に複数チームは不可 |
| ネストなし | チームメイトはさらにチームを作れない |
| リーダー固定 | リーダーの交代は不可 |
