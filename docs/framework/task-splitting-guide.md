# タスク分割ガイドライン

**作成日**: 2026-02-16
**目的**: 大規模タスクを適切に分割し、タイムアウトとレート制限を防止

---

## 1. 分割が必要なタスクの判断基準

### 1.1 分割必須の条件

| 条件 | 理由 |
|------|------|
| 推定実行時間 > 10分 | タイムアウトリスク |
| 対象ファイル > 10件 | コンテキスト過負荷 |
| 複数の独立した成果物 | 並列実行可能 |
| 複数ドメインの知識が必要 | 専門エージェント分担 |

### 1.2 分割不要の条件

| 条件 | 理由 |
|------|------|
| 単一ファイルの修正 | オーバーヘッドが大きい |
| 強い依存関係 | 分割メリットなし |
| 5分以内で完了 | 分割コスト > 効果 |

## 2. 分割パターン

### 2.1 機能分割（Functional Split）

**適用場面**: 異なる機能を持つサブタスク

```
元タスク: PRDの包括的レビュー
    ↓
分割後:
├── Task A: セキュリティ観点レビュー
├── Task B: 設計観点レビュー
├── Task C: 法規制観点レビュー
└── Task D: 結果統合（A, B, C完了後）
```

**Wave構成**:
```
Wave 1: [A, B, C]  # 並列実行
Wave 2: [D]        # 統合
```

### 2.2 データ分割（Data Split）

**適用場面**: 同じ処理を異なるデータに適用

```
元タスク: 全テーブル（10件）のスキーマレビュー
    ↓
分割後:
├── Task A: テーブル1-3のレビュー
├── Task B: テーブル4-6のレビュー
├── Task C: テーブル7-10のレビュー
└── Task D: 結果統合
```

**分割サイズ目安**:
- ファイル: 3-5件/タスク
- 行数: 500-1000行/タスク
- テーブル: 3-4件/タスク

### 2.3 フェーズ分割（Phase Split）

**適用場面**: 順序依存のある処理

```
元タスク: 新機能の設計から実装まで
    ↓
分割後:
├── Phase 1: 要件分析
├── Phase 2: 設計（Phase 1完了後）
├── Phase 3: 実装（Phase 2完了後）
└── Phase 4: テスト（Phase 3完了後）
```

**Wave構成**:
```
Wave 1: [Phase 1]
Wave 2: [Phase 2]
Wave 3: [Phase 3]
Wave 4: [Phase 4]
```

### 2.4 専門性分割（Expertise Split）

**適用場面**: 異なる専門知識が必要

```
元タスク: AppSheetアプリの設計レビュー
    ↓
分割後:
├── mutsubi-designer: UI/UX観点
├── mutsubi-engineer: 技術観点
├── mutsubi-researcher: 業務要件観点
└── mutsubi-integrator: 統合レビュー
```

## 3. 分割サイズの目安

### 3.1 時間ベース

| 元タスク推定時間 | 分割数 | 各タスク目標時間 |
|-----------------|--------|-----------------|
| 10-20分 | 2-3 | 5-7分 |
| 20-40分 | 4-6 | 5-7分 |
| 40分以上 | 6-8 | 5-7分 |

### 3.2 複雑度ベース

| 複雑度 | 分割粒度 | 例 |
|--------|---------|-----|
| 低 | 粗い（3-4タスク） | 定型作業の並列化 |
| 中 | 標準（5-6タスク） | 機能実装 |
| 高 | 細かい（7-8タスク） | アーキテクチャ設計 |

---

## 4. 依存関係の管理

### 4.1 依存関係の種類

| 種類 | 記号 | 説明 |
|------|------|------|
| 完了依存 | A → B | AがBより先に完了 |
| データ依存 | A ⇒ B | AのOutputがBのInput |
| 独立 | A ‖ B | 並列実行可能 |

### 4.2 依存関係の記述例

```
## Wave構成

Wave 1（並列）:
- Task A: PRDセクション1-3レビュー
- Task B: PRDセクション4-6レビュー
- Task C: PRDセクション7-9レビュー

Wave 2（Wave 1完了後）:
- Task D: 結果統合
  - 入力: A, B, Cの出力
  - 出力: 統合レビューレポート
```

### 4.3 ブロッキング回避

| 問題 | 対策 |
|------|------|
| 1タスクの遅延が全体に波及 | クリティカルパスを最小化 |
| 統合タスクのボトルネック | 統合タスクを軽量に設計 |
| 依存解決待ち | 独立タスクを先にスケジュール |

## 5. 分割チェックリスト

### タスク分析時

```
□ 推定実行時間を見積もったか
□ 対象ファイル数を確認したか
□ 独立して実行可能な部分を特定したか
□ 依存関係を明確にしたか
```

### 分割設計時

```
□ 各タスクが5-7分で完了するサイズか
□ 並列実行可能なタスクをWaveにグループ化したか
□ 統合タスクを用意したか
□ エラー時の影響範囲を限定できているか
```

### 実行時

```
□ Wave順序を守っているか
□ 並列数は5以下か
□ 各タスクの出力先パスを明示したか
□ 統合タスクに必要な入力を明記したか
```

## 6. アンチパターン

### 6.1 過度な分割

```
❌ 悪い例:
1行の修正を10タスクに分割
→ オーバーヘッドが効果を上回る

✅ 良い例:
関連する修正をまとめて1タスクに
```

### 6.2 不適切な依存関係

```
❌ 悪い例:
Wave 1: [A]
Wave 2: [B] (Aに依存)
Wave 3: [C] (Bに依存)
Wave 4: [D] (Cに依存)
→ 実質直列実行、分割の意味なし

✅ 良い例:
Wave 1: [A, B, C] (独立)
Wave 2: [D] (統合)
```

### 6.3 統合タスクの肥大化

```
❌ 悪い例:
統合タスクで全ての調整・修正を実行
→ 統合タスクがボトルネック化

✅ 良い例:
統合タスクは結果のマージのみ
追加修正は別タスクで
```
