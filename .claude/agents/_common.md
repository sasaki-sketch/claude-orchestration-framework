## 稼働ログフォーマット

**重要**: 稼働ログはオーケストレーターが管理する。エージェントは自分で時刻を記録しない。

### オーケストレーター用フォーマット

```markdown
## Wave {番号}: {Wave名} - {日付}

| エージェント | タスク | 定義参照 | 成果物 | ステータス |
|-------------|--------|---------|--------|----------|
| {project}-XX | タスク概要 | ✅/❌ | ファイルパス | 完了/失敗 |
| {project}-YY | タスク概要 | ✅/❌ | ファイルパス | 完了/失敗 |

**実行モード**: 並列 / 順次
**Wave開始**: HH:MM
**Wave完了**: HH:MM

---
```

### フィールド説明

| フィールド | 説明 | 記録者 |
|-----------|------|--------|
| Wave番号 | 実行グループ番号（1, 2, 3...） | オーケストレーター |
| Wave名 | 「PRDレビュー」「設計」等 | オーケストレーター |
| 実行モード | 並列/順次 | オーケストレーター |
| エージェント | {project}-pm, {project}-designer 等 | オーケストレーター |
| タスク | 実行したタスクの概要 | エージェント |
| 定義参照 | 定義ファイルを読んだか | エージェント |
| 成果物 | 作成・更新したファイルのパス | エージェント |
| ステータス | 完了/部分完了/失敗 | エージェント |

### エージェント用（プロンプトで指示）

エージェントには「タスク完了後、成果物のパスとステータスを報告してください」と指示する。
時刻・Wave番号はオーケストレーターが管理するため、エージェントは記録しない。

## 要件定義フェーズの必須ルール

1. **依頼者とのすり合わせ必須**: PRD作成後、実装前に主要ポイントを依頼者と確認する
2. **AskUserQuestion活用**: 必須/任意、スコープなど重要な判断は構造化された質問で確認
3. **元設計の意図尊重**: 既存フォームの「任意/必須」設定には業務上の理由がある。安易に変更しない
4. **ドキュメント一貫性**: 要件変更時はPRD・実装ガイド・テストケースすべてを更新する

## ワークフロー（4 Phase）

```
Phase 1: 設計
  └── designer が帳票・フロー設計、draft作成

Phase 2: レビュー
  └── reviewer が品質・法務チェック

Phase 3: 実装
  └── engineer が技術設計・実装

Phase 4: 検証
  └── reviewer が受け入れテスト
```

## レビュー議論（簡易版）

### フォーマット
```
論点: [タイトル]
指摘: [問題点]
対応: [解決策] or [保留理由]
結果: ✅合意 / ⚠️保留 / ❓要調査
```

### 分類（3カテゴリ）
| 結果 | 次アクション |
|-----|-------------|
| ✅合意 | 設計案に反映 |
| ⚠️保留 | ステークホルダー確認 |
| ❓要調査 | researcher/engineer検証 |
